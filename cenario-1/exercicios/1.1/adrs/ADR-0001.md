# ADR-0001: Escolha do Modelo de LLM para o Assistente de IA

## Status: Aceito

---

## Contexto

A NovaTech precisa de um assistente de IA integrado ao Microsoft Teams que responda perguntas em linguagem natural com base na documentação interna. O assistente utiliza arquitetura RAG (Retrieval-Augmented Generation): trechos de documentos recuperados são fornecidos ao LLM como contexto para cada resposta — o modelo não gera conhecimento livre, opera sobre o que o retrieval entrega.

**Volume estimado:**
- 320 chamados/dia × 60% = **192 queries/dia com consulta a documentos**
- Estimativa por query: ~4.000 tokens de input (system prompt + chunks recuperados + pergunta) e ~400 tokens de output
- Volume mensal: ~23M tokens de input e ~2,3M tokens de output

**Restrições do projeto:**
- Prazo de 3 meses (discovery + desenvolvimento + go-live)
- Integração com Microsoft Teams e SharePoint
- NovaTech já possui licenças Microsoft 365 E3 e Azure AI Services provisionados
- Requisito crítico: **o assistente nunca deve inventar informações**

**Forças em tensão:**
- Integração com ecossistema Microsoft vs. custo por token vs. capacidade de raciocínio vs. risco de compliance de dados
- Documentos contraditórios entre versões exigem que o modelo apresente ambas as versões com indicação de data, sem escolher uma — requer seguimento preciso de instrução estruturada
- ~15% dos documentos são escaneados (OCR necessário no pipeline) e PDFs com tabelas complexas — a qualidade de extração é independente do modelo

---

## Decisão

**Utilizaremos Claude Sonnet 4.6 (via API Anthropic) como modelo LLM**, sujeito à condição pré-requisito de assinatura de DPA (Data Processing Agreement) com a Anthropic antes do início do desenvolvimento. A camada de integração ao LLM será encapsulada em uma interface definida contratualmente (ver Consequências), tornando o modelo intercambiável.

---

## Cálculo de custo corrigido

Qualquer pipeline RAG para a base documental da NovaTech (~800 documentos SharePoint + ~400 páginas Confluence + planilhas) requer um vetor store. As opções gerenciadas custam ~$70–100/mês; a NovaTech já tem Azure AI Search provisionado. **O custo do Azure AI Search (~$73/mês no tier Basic para o volume estimado) é necessário independentemente do modelo escolhido** e não deve ser atribuído exclusivamente à opção Azure OpenAI.

| Opção | Tokens/mês | Azure AI Search | App Service / Functions | **Total/mês** |
|---|---|---|---|---|
| Azure OpenAI GPT-4o | ~$150 | $73 | $50 | **~$273** |
| Claude Sonnet 4.6 (Anthropic) | ~$104 | $73 | $50 | **~$227** |
| Claude Haiku 4.5 (Anthropic) | ~$35 | $73 | $50 | **~$158** |
| Ollama (GPU spot A10, 730h/mês) | — | $73 | ~$300–400 (GPU) | **~$450–550** |

A diferença entre Azure OpenAI e Claude Sonnet 4.6 é **$46/mês (~17%)**, não 44% como calculado anteriormente (que omitia o Search do custo de Claude). A economia de $46/mês ao longo de 12 meses é $552 — valor real mas não o argumento principal desta decisão.

---

## Decisão justificada

A escolha de Claude Sonnet 4.6 sobre Azure OpenAI GPT-4o apoia-se em dois argumentos principais, com os limites de cada um declarados explicitamente:

### 1. Raciocínio em documentos contraditórios

O requisito de apresentar duas versões de um mesmo procedimento com indicação de data — sem fundir as versões ou escolher uma — requer seguimento preciso de instrução de formatação estruturada. Em avaliações internas de Q&A factual com contexto fornecido, Claude Sonnet 4.6 demonstra comportamento mais conservador quando instruído a não extrapolar o contexto recebido.

**Limite deste argumento:** Não existe benchmark público específico para Q&A factual em RAG com documentos contraditórios em português do Brasil. A superioridade de Claude Sonnet sobre GPT-4o neste caso de uso específico é uma avaliação qualitativa, não uma medição. Ambos os modelos são capazes de cumprir o requisito se o prompt for bem construído. A escolha de Sonnet é conservadora, não definitiva.

**Nota importante:** Em sistemas RAG, a alucinação é primariamente uma propriedade do pipeline de retrieval, não do modelo. O que mais protege contra invenção de informações é o retrieval preciso (chunks relevantes e bem ranqueados), o prompt com instrução explícita de não extrapolar o contexto, e a citação obrigatória de fonte. Estas medidas se aplicam independentemente do modelo escolhido e devem ser implementadas na ADR-0002 (estratégia de contexto) e ADR-0004 (pipeline RAG).

### 2. Distribuição de risco entre fornecedores

Concentrar LLM e infraestrutura no mesmo fornecedor (Microsoft) cria dependência única de preço, SLA e disponibilidade. Claude Sonnet 4.6 distribui esse risco, com a interface abstraída permitindo retornar ao Azure OpenAI se necessário.

**Limite deste argumento:** A distribuição de risco só é real se a interface abstraída for implementada conforme especificado em Consequências. Sem essa implementação, o argumento é fictício.

## Consequências

### Positivas
- Custo mensal ~$46 menor que Azure OpenAI GPT-4o para o volume estimado
- Janela de contexto de 1M tokens com margem operacional para queries multi-domínio e crescimento da base documental
- Risco de fornecedor distribuído entre Anthropic (modelo) e Microsoft (infraestrutura, indexação, integração)

### Riscos e mitigações obrigatórias

**Risco 1 — Compliance e soberania de dados (CRÍTICO)**

Ao usar a API Anthropic, chunks de documentos e perguntas dos atendentes são processados em servidores Anthropic fora da infraestrutura Azure da NovaTech. A documentação pode conter cláusulas contratuais, SLAs de clientes e informações de compliance sujeitas a restrições de confidencialidade.

**Mitigação obrigatória (pré-requisito de implementação):**
1. Assinar DPA (Data Processing Agreement) com Anthropic antes de iniciar o desenvolvimento
2. Verificar com o jurídico da NovaTech se os contratos com clientes permitem processamento de dados em provedor terceiro fora do Azure
3. Confirmar que a Anthropic atende os requisitos de retenção zero (a API Anthropic não usa dados de produção para treinamento por padrão, mas isso deve ser formalmente documentado)

Se o jurídico vetar o processamento externo, a decisão deve ser revertida para Azure OpenAI GPT-4o, que opera dentro do contrato Microsoft 365 E3 já aprovado pela NovaTech.

---

**Risco 2 — Interface de abstração: implementação real, não promessa**

O ADR menciona encapsular o cliente LLM em uma interface `ILLMProvider`. Para que essa abstração seja real — e não apenas uma promessa no documento — o contrato da interface deve ser definido antes do início da implementação.

**Especificação mínima da interface:**

```
ILLMProvider:
  - chat(systemPrompt: string, userMessage: string, context: string[]) → LLMResponse
  - LLMResponse: { content: string, sources: string[], inputTokens: int, outputTokens: int }
  - Sem exposição de tipos específicos de provedor (sem AnthropicClient, sem OpenAIClient)
  - Gerenciamento de retry e rate limit: responsabilidade da implementação, não da interface
```

O pipeline RAG interage exclusivamente com `ILLMProvider`. A implementação `AnthropicProvider` e uma implementação stub `AzureOpenAIProvider` (com chamadas reais ou mock) devem existir desde o início para validar que a interface suporta troca real.

---

**Risco 3 — Latência de API externa**

A API Anthropic opera fora da VNet Azure da NovaTech. Em condições normais, a latência adicionada é de 100–300ms, irrelevante para o alvo de <2 minutos por consulta. Em degradação parcial da API Anthropic, o assistente pode ficar indisponível.

**Mitigação:**
- Timeout de 30s com retry exponencial (máximo 2 tentativas)
- Health check da API Anthropic no dashboard de monitoramento
- Runbook: se a API Anthropic estiver indisponível por mais de 15 minutos, notificar os atendentes via Teams e registrar incidente

---

**Risco 4 — Crescimento de volume**

Se o volume dobrar (384 queries/dia), o custo de tokens Claude Sonnet passa de $104 para ~$208/mês (total ~$331/mês), ultrapassando o custo atual do Azure OpenAI.

**Mitigação:** Configurar budget alert no Azure Cost Management para o gasto total do projeto ao atingir $300/mês. Reavaliação de modelo quando o volume ultrapassar 350 queries/dia.

## Alternativas consideradas

### Azure OpenAI (GPT-4o)
- **Prós:** Integração dentro do ecossistema Microsoft já aprovado pela NovaTech; Azure AI Search com Semantic Ranker e groundedness score nativos; DPA coberto pelo contrato E3 existente; sem novo fornecedor para aprovação jurídica
- **Contras:** Custo $46/mês maior; concentração total de risco no ecossistema Microsoft; Azure AI Search resolve anti-alucinação na camada de retrieval (não elimina a vantagem arquitetural de distribuir fornecedores)
- **Razão do descarte:** A economia de $46/mês e a distribuição de risco justificam a adição de um segundo fornecedor, desde que o DPA seja assinado. Se o DPA não for viável, esta opção é a alternativa direta.

### Claude Haiku 4.5
- **Prós:** Custo total ~$158/mês — $69/mês mais barato que Sonnet
- **Contras:** Janela de contexto de 200K tokens (sem margem para crescimento); desempenho inferior em seguimento de instruções estruturadas complexas (necessário para apresentar versões contraditórias de documentos); maior risco de respostas incompletas em queries multi-domínio
- **Razão do descarte:** O requisito de documentos contraditórios é o caso mais difícil do projeto. Usar o modelo mais econômico no caso mais crítico inverte a prioridade de risco. A economia de $69/mês não justifica o risco.

### Ollama (self-hosted, Llama 3.3 70B)
- **Prós:** Dados nunca saem da infraestrutura; sem custo por token
- **Contras:** Custo total estimado $450–550/mês (GPU spot + Azure AI Search + infra); complexidade operacional (serving de modelo, escalonamento, disponibilidade) é escopo adicional não previsto no projeto de 3 meses; qualidade inferior a modelos gerenciados de mesmo tier de preço
- **Razão do descarte:** Mais caro e mais arriscado para o prazo. Considerar apenas se requisitos de soberania de dados vetarem qualquer API externa.

## Histórico de revisão

**Proposta inicial:** Azure OpenAI GPT-4o, com argumento de integração nativa e custo estimado de $150/mês.

**Revisão 1 (devil's advocate, fase interna):** Contra-argumentos identificaram que o custo real do Azure OpenAI incluía Azure AI Search ($73/mês) e App Service ($50/mês), totalizando $273/mês. Isso motivou a reavaliação para Claude Sonnet 4.6 como opção de menor custo.

**Revisão 2 (devil's advocate, fase 2):** Contra-argumentos identificaram que:
1. O cálculo de custo anterior aplicava Azure AI Search apenas ao Azure OpenAI, inflando artificialmente a diferença. Corrigido: a diferença real é $46/mês (17%).
2. O risco de compliance e soberania de dados (DPA Anthropic) não havia sido registrado. Adicionado como pré-requisito crítico.
3. O argumento de anti-alucinação baseado em modelo não tem evidência concreta para o caso de uso específico. Corrigido: o argumento foi enfraquecido e reancora a responsabilidade de anti-alucinação na arquitetura do pipeline (retrieval + prompt), não no modelo.
4. A interface `ILLMProvider` estava descrita como promessa sem especificação. Adicionada especificação mínima de contrato.

**Decisão mantida** com as correções acima. Se o DPA Anthropic não for aprovado pelo jurídico da NovaTech, reverter para Azure OpenAI GPT-4o sem necessidade de nova ADR.
