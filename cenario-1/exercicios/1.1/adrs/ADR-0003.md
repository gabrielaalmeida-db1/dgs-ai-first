# ADR-0003: Tratamento de Documentos Contraditórios

## Status: Aceito

---

## Contexto

A NovaTech tem três áreas (Operações, Compliance, Comercial) que atualizam documentos de forma independente, sem processo unificado de revisão. Versões antigas coexistem com versões novas sem invalidação formal — já foram identificadas pelo menos 3 contradições explícitas. O risco concreto: o assistente pode recuperar dois chunks sobre o mesmo procedimento com regras diferentes (ex: "prazo de 5 dias úteis" vs "prazo de 7 dias corridos") e gerar uma resposta que mistura as duas versões silenciosamente. O requisito é claro: o assistente nunca deve escolher uma versão sem avisar o atendente de que existe outra.

---

## Decisão

### Camada 1 — Detecção: híbrida (metadado automático + lista curada)

O pipeline identifica contradições de dois modos:

**Modo A — Automático por metadado (mesma série documental):**
Quando um novo documento é ingerido, o pipeline verifica se já existe no índice um documento com o mesmo `document_code`. Se sim, o novo recebe `status: "ativo"` e o anterior recebe `status: "substituído"` e `contradiction_flag: true`. A relação entre os dois é armazenada em `contradiction_pair_id`.

**Modo B — Lista curada (contradições entre documentos de áreas diferentes):**
O responsável documental designado pela NovaTech mantém o arquivo `contradiction_registry.json` com os pares de documentos que se contradizem entre áreas. A entrada no registro define manualmente `contradiction_flag: true` e o `contradiction_pair_id` em ambos os documentos. Os 3 casos já identificados entram neste registro antes do go-live.

> **Pré-requisito de go-live:** a NovaTech deve nomear o responsável pelo `contradiction_registry.json` antes do início da implementação. Se não houver responsável designado, o sistema entra em produção com apenas o Modo A ativo e os 3 casos conhecidos mapeados via `document_code`.

**Schema de metadados por documento:**

| Campo | Tipo | Valores possíveis | Quem preenche |
|---|---|---|---|
| `document_code` | string | ex: `"OPS-021"`, `"COMP-007"` | Pipeline de ingestão, a partir do nome/propriedade do arquivo |
| `document_date` | date | data do documento original | Pipeline de ingestão |
| `area` | string | `"operacoes"`, `"compliance"`, `"comercial"` | Pipeline de ingestão, a partir da pasta de origem |
| `status` | string | `"ativo"`, `"substituído"`, `"em_revisao"` | Pipeline automático (Modo A) ou registro curado (Modo B) |
| `contradiction_flag` | boolean | `true`, `false` | Pipeline automático (Modo A) ou registro curado (Modo B) |
| `contradiction_pair_id` | string ou null | ID do documento contraditório | Pipeline automático (Modo A) ou registro curado (Modo B) |

**Detecção semântica descartada para go-live:** alta taxa de falso positivo, requer semanas de ajuste de threshold. Entra como item de roadmap pós-go-live: chunks com similaridade coseno > 0,92 entre documentos de áreas diferentes, mas com valores numéricos divergentes, serão marcados como `status: "em_revisao"` e enviados para revisão humana.

---

### Camada 2 — Armazenamento: indexar ambas as versões com metadados e flag de contradição

Ambos os documentos ficam no índice. O documento mais antigo não é removido — ele recebe `status: "substituído"` mas permanece disponível. Quando o retrieval encontra um chunk com `contradiction_flag: true`, ele obrigatoriamente também busca o chunk do `contradiction_pair_id` correspondente. Os dois são entregues ao LLM como um par, ocupando juntos **1 slot dos 5 disponíveis** (ADR-0002) — não 2 slots separados.

**Indexar apenas a versão mais recente foi descartado:** viola o requisito explícito de exibir ambas as versões com data de vigência. Além disso, a versão antiga pode estar em vigor para contratos anteriores à atualização.

**Indexar sem contradiction_flag foi descartado:** sem o flag, o retrieval não sabe que precisa buscar o par — o segundo documento só chegaria ao LLM por acaso, se fosse semanticamente relevante para a query.

---

### Camada 3 — Apresentação: ambas as versões rotuladas, com hierarquia de área, sem escolha pelo LLM

Quando o LLM recebe um par com `contradiction_flag: true`, a instrução no system prompt é:

```
Quando você receber chunks marcados como documentos em contradição,
use obrigatoriamente este formato:

⚠️ Conflito entre versões identificado

📄 Versão mais recente — [título], [área], [data]:
[conteúdo do chunk mais recente]

📄 Versão anterior — [título], [área], [data]:
[conteúdo do chunk mais antigo]

ℹ️ Existem duas versões em vigor. A versão mais recente é de [data].
Fonte normativa para este tema: [Compliance para procedimentos
regulatórios | Comercial para regras de contrato | Operações
para procedimentos de execução].
Confirme com seu gestor qual versão aplicar antes de informar o cliente.

Regras absolutas:
- Nunca omita uma das versões.
- Nunca escolha qual versão é a correta para o caso do cliente.
- Nunca misture conteúdo das duas versões em uma única resposta.
```

O LLM indica qual versão é mais recente (factual) e qual área é a fonte normativa para o tema (critério de hierarquia). Não recomenda qual versão aplicar ao caso concreto — essa decisão é do atendente e do gestor.

**Delegar a decisão ao LLM foi descartado:** o modelo não tem acesso ao contrato do cliente, ao período de transição entre versões ou às exceções operacionais. Erros em compliance de carga ou SLA têm consequência contratual real — a responsabilidade não pode ficar com um modelo de linguagem.

**Mostrar apenas a mais recente foi descartado:** viola o requisito explícito e esconde informação que o atendente precisa para entender que há ambiguidade.

---

## Por que essa decisão

**Por que detecção híbrida e não só automática:**
Contradições entre documentos de áreas diferentes (ex: Compliance e Comercial com regras conflitantes sobre o mesmo procedimento) nunca terão o mesmo `document_code` — o modo automático não as captura. A lista curada resolve os casos conhecidos; o modo automático resolve as atualizações de versão dentro da mesma série.

**Por que indexar ambas:**
O requisito não é opcional. Além disso, a versão antiga pode estar em vigor para contratos firmados antes da atualização — o atendente precisa saber que ela existe.

**Por que o LLM não escolhe, mas indica a hierarquia:**
O LLM não tem contexto do caso específico do cliente. Mas indicar qual área é a fonte normativa para o tema (Compliance > Comercial para questões regulatórias, por exemplo) dá ao atendente um critério de decisão objetivo sem que o modelo assuma responsabilidade pelo caso concreto.

---

## Consequências

**Fica mais simples:**
- O pipeline de ingestão tem comportamento claro: mesmo `document_code` → set `status` e `contradiction_flag` automaticamente
- O desenvolvedor sabe exatamente o que colocar no system prompt
- O atendente nunca recebe uma resposta que mistura versões sem aviso

**Exige manutenção contínua:**
- Alguém na NovaTech precisa ser designado antes do go-live como responsável pelo `contradiction_registry.json` e treinado para atualizar quando novos conflitos entre áreas aparecerem
- Quando a NovaTech resolver uma contradição (publicar versão unificada), o registro precisa ser atualizado manualmente — o sistema detecta conflito, não resolução

**O que o atendente vai experimentar:**
- Para ~95% das perguntas: resposta normal com citação de fonte
- Para perguntas que tocam os documentos contraditórios: aviso visual com as duas versões, indicação da área normativa e instrução de confirmar com o gestor
- O assistente vai parecer "indeciso" nesses casos — isso é intencional e correto

**Precisa ser monitorado:**
- Volume de queries que disparam o aviso de contradição (crescimento rápido = novos conflitos não curados)
- Feedback dos atendentes: se o aviso aparece com frequência alta, indica que o registro de contradições está desatualizado
- Pós-go-live: implementar detecção semântica de contradições não mapeadas (chunks com similaridade > 0,92 e valores numéricos divergentes → `status: "em_revisao"`)

## Histórico de devil's advocate

**Contra-argumento 1 — Lista curada será abandonada sem responsável designado**
A NovaTech não tem processo de revisão unificado — pedir que mantenha um registro adicional é criar dependência sem garantia de execução. *Resposta:* Adicionado como pré-requisito de go-live: responsável documental nomeado antes do início da implementação. Sem nomeação, o sistema entra com apenas detecção automática por `document_code`.

**Contra-argumento 2 — Contradições não mapeadas criam falsa sensação de segurança**
O `contradiction_flag` só cobre pares conhecidos. Novos conflitos não curados chegam ao LLM sem flag, gerando respostas que misturam versões sem aviso. *Resposta:* Adicionado item de roadmap pós-go-live: detecção semântica de candidatos a contradição via threshold de similaridade, marcando automaticamente como `em_revisao` para revisão humana.

**Contra-argumento 3 — Mostrar duas versões sem critério é inútil durante o atendimento**
O atendente em chamada não pode pausar para consultar o gestor a cada contradição. *Resposta:* Adicionada hierarquia de área normativa no system prompt (Compliance > Comercial para questões regulatórias, etc.). O LLM indica qual área é a fonte normativa para o tema — isso dá um critério de decisão ao atendente sem que o modelo escolha pelo caso concreto.
