# ADR-0004: Build vs Buy para o Pipeline de RAG

## Status: Aceito

---

## Contexto

O projeto precisa de uma infraestrutura que indexe ~1.200 documentos de três fontes (SharePoint, Confluence, planilhas), atualize o índice em até 24h, suporte os metadados de contradição do ADR-0003, e entregue até 5 chunks relevantes por query ao Claude Sonnet 4.6. A decisão afeta custo operacional, velocidade de implementação e o quanto o time da DB1 vai precisar operar e depurar em produção. A restrição principal é o prazo de 3 meses — qualquer componente que exija semanas de configuração antes de funcionar é um risco real.

---

## Decisão

**Combinação escolhida: Azure AI Search (retrieval) + código customizado Python (orquestração) + Azure Document Intelligence (extração de PDFs e OCR)**

### Vetor store: Azure AI Search (tier Basic)

Configuração do índice:

| Campo | Tipo | Uso |
|---|---|---|
| `id` | string | identificador único do chunk |
| `content` | string | texto do chunk (até 1.000 tokens) |
| `embedding` | vector(1536) | busca semântica |
| `document_code` | string | filtro para contradições (ADR-0003) |
| `document_title` | string | citação de fonte na resposta |
| `area` | string | filtro por domínio (ADR-0002) |
| `status` | string | `ativo`, `substituído`, `em_revisao` |
| `contradiction_flag` | boolean | trigger para busca do par |
| `contradiction_pair_id` | string | ID do chunk contraditório |
| `document_date` | date | ordenação por versão |
| `source` | string | `sharepoint`, `confluence`, `planilhas` |

O retrieval executa: busca semântica com filtro `status eq 'ativo'`. Se algum resultado tiver `contradiction_flag: true`, executa segunda busca por `contradiction_pair_id` — os dois chunks chegam juntos ao LLM como 1 slot (ADR-0002).

### Extração de documentos: Azure Document Intelligence

Responsável por PDFs com tabelas complexas e os ~15% de documentos escaneados (OCR). Output: texto estruturado por seção → entrada para o chunker de 1.000 tokens. Já incluído no Azure AI Services provisionado pela NovaTech — sem custo adicional de licença.

**Fallback obrigatório:** documentos com taxa de extração abaixo de 80% de caracteres legíveis são sinalizados como `status: "em_revisao"` e não entram no índice automaticamente. O responsável documental da NovaTech revisa manualmente antes da indexação.

### Orquestração: código customizado Python

O pipeline de query tem 7 passos lineares:

```
1. receber query do Teams bot
2. detectar domínios mencionados (palavras-chave — ADR-0002)
3. buscar no Azure AI Search com diversificação por domínio
4. verificar contradiction_flag e buscar pares se necessário
5. montar prompt dentro do limite de 8.000 tokens (ADR-0002)
6. chamar Claude Sonnet 4.6 via Anthropic SDK
7. retornar resposta com citações ao Teams bot
```

Esse pipeline cabe em ~400 linhas de Python sem abstração de framework.

### Pipeline de atualização (requisito de 24h)

Azure Function com timer trigger a cada 4h:

1. Microsoft Graph API lista documentos modificados no SharePoint desde a última execução
2. Confluence API lista páginas modificadas (implementação customizada — ver Consequências)
3. Azure Document Intelligence processa PDFs e documentos escaneados novos/modificados
4. Chunker de 1.000 tokens com overlap de 10% divide o conteúdo extraído
5. Embedding gerado e chunk indexado no Azure AI Search com metadados do ADR-0003
6. Documentos com `contradiction_flag: true` disparam verificação do par no índice

## Por que essa decisão

### Vetor store: Azure AI Search, não ChromaDB/FAISS

O Azure AI Search entrega três vantagens independentes do LLM escolhido:

**1. Conector nativo do SharePoint (Microsoft Graph API):** economiza 2-3 semanas de implementação de pipeline de ingestion customizado. FAISS e ChromaDB não têm esse conector.

**2. Filtros compostos por metadados:** suporte nativo a `status eq 'ativo' AND area eq 'compliance'` com eficiência de índice invertido — necessário para ADR-0002 e ADR-0003. ChromaDB suporta filtros mas com performance menor em queries compostas.

**3. Serviço gerenciado:** sem backups, escalonamento ou upgrades manuais. O time DB1 não tem histórico de operar vetores self-hosted em produção.

**"Azure AI Search perde valor sem Azure OpenAI" é incorreto:** o serviço é infraestrutura de busca. O Semantic Ranker, os filtros e o conector SharePoint funcionam com qualquer LLM downstream — incluindo Claude Sonnet 4.6 via Anthropic.

**ChromaDB self-hosted descartado:** custo similar ($50-70/mês de container) sem conector nativo do SharePoint; adiciona operação que o time não tem histórico de gerenciar em produção.

**FAISS descartado:** biblioteca embutida no processo, sem suporte a atualização incremental em 24h sem reescrita do índice inteiro; sem filtros por metadados nativos.

### Orquestração: código customizado, não frameworks

O pipeline de query tem 7 passos lineares sem ramificações, sem agentes, sem memória de longa duração. LangChain e LlamaIndex são justificados quando o pipeline tem complexidade que não vale implementar do zero. Nenhum desse caso se aplica aqui.

**Custo real de usar LangChain:**
- A lógica de `contradiction_flag` do ADR-0003 exige controle fino sobre quais chunks entram no contexto — isso luta contra as abstrações de retrieval do LangChain, que não foi projetado para "se esse chunk tiver flag X, busque também o chunk Y"
- LangChain tem histórico de breaking changes em versões minor (0.1→0.2→0.3)
- Depurar uma resposta errada num pipeline LangChain exige entender a cadeia de abstrações; em código puro, o log mostra o prompt exato enviado ao LLM

**LangChain descartado:** abstrações que conflitam com a lógica de contradição do ADR-0003; histórico de instabilidade de API.

**LlamaIndex descartado para orquestração:** os conectores de documento são úteis, mas o Azure Document Intelligence cobre OCR e extração de tabelas com melhor suporte ao formato de documentos da NovaTech; o restante do framework não justifica a dependência.

**Azure Prompt Flow descartado:** não suporta Claude Sonnet 4.6 nativamente (projetado para Azure OpenAI); a customização necessária para ADR-0003 exigiria um plugin externo que negaria qualquer vantagem do low-code.

### O argumento "NovaTech já tem Azure": vantagem real, não sunk cost

É vantagem real para componentes específicos — não para o ecossistema completo:

- A **fonte de dados principal** (SharePoint) é Azure. O conector nativo não é sunk cost — é redução concreta de escopo.
- O **pipeline de atualização** usa Microsoft Graph API de qualquer forma. Rodar a Azure Function no mesmo tenant simplifica autenticação via Managed Identity, sem gestão de secrets.
- O **Azure Document Intelligence** já está no Azure AI Services provisionado — zero custo adicional de licença para o volume inicial.

O que **não** justifica a escolha Azure: o argumento de "integração com Azure OpenAI" — ADR-0001 escolheu Claude Sonnet 4.6 via Anthropic API. Esse argumento não se aplica a este projeto.

## Consequências

**Fica mais fácil:**
- Ingestion do SharePoint: conector oficial + Managed Identity, sem código de autenticação customizado
- OCR e tabelas complexas: Azure Document Intelligence já provisionado e testado para documentos Office/PDF
- Operação: nenhum serviço adicional para gerenciar além do Azure AI Search (serviço gerenciado)
- Debugging: código Python direto, sem abstrações de framework — o log mostra o prompt exato enviado ao Claude

**Exige trabalho adicional:**
- **Conector do Confluence (estimativa: 1-2 semanas):** não tem suporte nativo no Azure AI Search. Precisa ser implementado via Confluence REST API — autenticação, paginação, extração de conteúdo de páginas e espaços. Este é o maior risco de escopo do ADR.
- Chunker de 1.000 tokens com lógica de overlap manual
- Lógica de diversificação por domínio do ADR-0002
- Segunda busca para pares de contradição do ADR-0003
- Fallback manual para documentos com extração abaixo de 80% de qualidade

**O que o time precisa saber operar:**
- Azure AI Search: criar e atualizar índice, monitorar quotas do Semantic Ranker (~$5/mês adicional para o volume estimado)
- Azure Document Intelligence: monitorar falhas de OCR via alerta de `status: "em_revisao"`
- Azure Functions: monitorar o timer trigger de 4h; alertar se o pipeline de atualização falhar

**Custo mensal estimado (total do projeto):**

| Componente | Custo/mês |
|---|---|
| Claude Sonnet 4.6 tokens (ADR-0001) | ~$138 |
| Azure AI Search Basic | $73 |
| Azure Functions (pipeline de query + atualização) | ~$20 |
| Azure Document Intelligence (~500 páginas/mês de atualizações) | ~$5 |
| **Total** | **~$236/mês** |

Dentro do limite de $300/mês. Margem de $64/mês para crescimento de volume.

## Histórico de devil's advocate

**Contra-argumento 1 — Azure AI Search a $73/mês entrega pouco sem Azure OpenAI**
O Semantic Ranker no tier Basic tem quota gratuita de 1.000 queries/mês; para 5.760 queries/mês, o custo adicional é ~$4,76/mês — irrelevante. O real diferenciador a $73/mês é o conector SharePoint + filtros por metadados + serviço gerenciado. ChromaDB num container Azure custaria ~$50/mês sem o conector.
*Resposta:* A economia de $23/mês não justifica implementar pipeline de ingestion SharePoint do zero (2-3 semanas de desenvolvimento). O argumento do sunk cost não se aplica porque o conector SharePoint tem valor real independente do LLM.

**Contra-argumento 2 — Código customizado subestima os casos difíceis da ingestion**
PDFs com tabelas quebradas em múltiplas páginas, documentos escaneados com qualidade ruim, planilhas com células mescladas — LlamaIndex tem conectores testados para esses casos com fallbacks prontos. Código customizado significa implementar e depurar cada caso durante os 3 meses.
*Resposta:* Adicionado fallback explícito: documentos com taxa de extração < 80% são sinalizados como `em_revisao` e não entram no índice automaticamente. O Azure Document Intelligence cobre os casos mais difíceis melhor do que LlamaIndex para documentos Microsoft Office.

**Contra-argumento 3 — O conector do Confluence é um buraco que o ADR ignorou**
O argumento central do Azure AI Search é o conector nativo do SharePoint — mas as ~400 páginas do Confluence exigem uma implementação customizada de qualquer forma. A vantagem do "conector nativo" cobre apenas metade da base documental.
*Resposta:* Adicionado ao ADR com estimativa explícita de 1-2 semanas de implementação. A comparação correta é: Azure AI Search + conector customizado Confluence vs LlamaIndex + ChromaDB. A diferença de $23/mês ainda não justifica trocar o stack — mas o esforço do conector Confluence precisa estar no planejamento do projeto, não como detalhe a resolver depois.
